1. Data generation

Household data is scenario-driven and distributionally checked

The generator creates affluent US advisor-like households rather than generic population samples. Scenarios are explicit, weighted, and validated for coverage. Public-data anchors and curated priors drive income, assets, liabilities, household structure, risk tolerance, and lifecycle fields.

Coverage requirement

11 scenarios

Every household belongs to one named archetype. This avoids hidden rare cases and makes coverage auditable.

Affluent conditioning

RIA-like population

Income and wealth distributions are shifted upward and constrained by an affluent income floor.

Validation

0 rule violations

The generated dataset has no business-rule violations in the current artifact set.

2. Dialog generation

Dialogs are generated in stages and checked for grounding and realism

Dialog generation does not ask a model to produce an entire transcript in one shot. It builds a persona, an outline, phase-level dialog turns, and state updates, then optionally finalizes the transcript into a more natural conversation.

Grounded transcript construction

The dialog generator consumes the household financial profile and scenario context. It writes a transcript, JSON metadata, evidence items, validation metrics, and plain-text transcript files.

Evidence artifacts map profile fields to advisor/client excerpts. These evidence files later become dialog-grounded GT for extraction scoring.

DeepSeek realism gate

A DeepSeek judge scores finalized transcripts on a 1 to 5 realism scale. The standard gate copies transcripts into a `realism_passed` subset when the realism score meets the configured threshold.

In the Train set, 316 dialogs are from the realism-passed subset. For the holdout test, the requested evaluation used 99 valid generated dialogs with judge artifacts retained.

Why this matters

The extraction benchmark is only meaningful if the dialogs sound like plausible financial-discovery conversations and if each scored field can be traced back to information present in the transcript. The realism gate helps with conversational quality; evidence-based grounded GT helps with fair scoring.

3. Matching and error accounting

Metrics are computed after record matching and grounded-aware field scoring

The evaluator compares extracted JSON to dialog-grounded GT. It normalizes both sides, pairs records by content when IDs are unstable, then scores fields cell-by-cell.

Record matching

Primary keys are used when reliable. Otherwise, content-based pairing is used for people, income lines, assets, liabilities, and protection policies.

Field scoring

Numeric fields use 1% relative tolerance. Multichoice fields are compared as sets. String-like fields are compared case-insensitively after normalization.

Error types

Errors are split into missing extracted values, invented extra values, value mismatches, and record-pairing failures.

Scoring assumptions

Assumption Implementation Why it matters
Dialog-grounded denominator When `ground_truth_is_grounded=true`, only fields present in grounded GT enter the denominator. Extractor is not penalized for facts never stated in the dialog.
ID exclusion Primary keys and `_id` fields are excluded by default. We evaluate semantic extraction, not synthetic key guessing.
Derived ratio exclusion Fields ending in `_ratio` are excluded. Ratio fields are derived and would otherwise double-count numeric facts.
PII/person-detail exclusions Configured in `03_data_extraction/config/scoring_exclusions.json`. Reduces noise from fields that are not central to financial extraction quality.
4. Train and holdout test logic

Train was used for pipeline development; holdout test is a held-out evaluation

The 316 Train dialogs were used while debugging the extraction pipeline, prompts, scoring, and discrepancy analysis. The 99 holdout test dialogs were generated from households that did not overlap with the Train extracted households and were not used for extractor/evaluator tuning.

6. Quality controls

Each stage has its own checks before the next stage consumes its outputs

Quality is controlled as a pipeline property, not only as a final accuracy number. We validate the generated household data, the dialog generation process, the dialog-grounded GT construction, extraction behavior on Train, and the full pipeline on the holdout test.

1. Synthetic data quality

The household generator is checked with business rules that enforce relationships between fields: lifecycle date constraints, parent/child age consistency, mortgage burden caps, alimony-marital-status compatibility, and future final-payment dates for non-revolving liabilities.

Statistical tests compare the generated data to priors using distributional diagnostics such as PSI and Jensen-Shannon divergence. Anomaly detection is run with two complementary models: an autoencoder reconstruction score and an IsolationForest score.

2. Dialog generation quality

Dialogs are generated in pieces rather than as a single monolithic response: persona, outline, phase-by-phase transcript, state updates, and optional final polishing. This reduces drift and makes the conversation easier to validate incrementally.

Evidence artifacts map profile facts to transcript excerpts. The generation pipeline also writes metrics and uses validation checks, including regex-style and normalization-based checks, to catch fields that are not properly grounded in the transcript. A DeepSeek realism judge adds a separate conversational-quality gate.

3. Ground truth quality

Extraction scoring uses dialog-grounded GT whenever evidence is available. This is a sparse GT profile built only from fields that have evidence in the transcript, so the evaluator does not ask the extractor to recover unspoken facts from the full synthetic profile.

The GT export reports how many dialogs were seen, how many profiles were written, how many evidence items were used, and whether any dialogs were missing profile metadata or household identifiers. During holdout evaluation, the pipeline confirmed 99 dialogs written, 99 grounded profiles used, and no missing profile/text rows.

4. Extraction quality on Train

The 316-dialog Train set was used to debug the extractor, prompts, normalization, rescue passes, record matching, and error reporting. This is where we iterated on the pipeline and inspected discrepancy tables to identify systematic extraction failures.

Train quality is monitored through extracted status reports, coercion issues, coverage aggregates, field-level discrepancy stats, value-mismatch tables, and accuracy histograms.

5. End-to-end pipeline quality on the holdout test

The holdout test validates the complete pipeline after tuning: held-out dialog generation, grounded GT export, extraction, record matching, scoring, discrepancy analysis, and final metrics. The holdout households do not overlap with the 316 Train households used during pipeline development.

This final gate is the closest approximation to an honest out-of-sample test: the same code and thresholds are applied, but the 99 dialogs were not part of extractor/evaluator iteration.

7. Next steps

Research and product directions

The current benchmark gives us a controlled end-to-end baseline. The next phase should make the dialogs more realistic, stress the extractor under harder conditions, and connect extraction errors to real product outcomes.

1. Improve dialog realism with multi-model critique

Send generated dialogs to several model families such as Grok, Qwen, DeepSeek, GPT, Gemini, Claude, and others. Ask each model for a concrete action plan for improving realism, aggregate feedback across dialogs and models, and iteratively update the dialog-generation prompts.

2. Run noise-based stress tests

Inject controlled noise into dialogs and measure how extraction metrics degrade. Examples include typos, transcript artifacts, repeated turns, missing punctuation, speaker-label noise, irrelevant side conversations, partial truncation, and contradictory corrections.

3. Build adversarial dialog attacks

Train or prompt an adversarial model to make minimal dialog edits that break the extractor. Use these attacks to design defenses, stronger validation, or a more robust extraction architecture.

4. Compare structured output modes

Compare extraction with and without structured generation / structured output constraints. Measure not only accuracy, but also parse failures, coercion issues, latency, retry rate, and cost.

5. Study dialog length versus extraction quality

Quantify how transcript length affects extraction quality and latency. Identify the dialog size at which extraction should switch from single-pass processing to batching, chunking, or hierarchical extraction.

6. Add a pre-enrichment mode

Improve client experience by pre-filling as much information as possible from public or permissioned sources, such as public professional profiles or other available records. The dialog can then focus only on missing, ambiguous, or stale facts, shortening the conversation while still refreshing outdated information.

7. Make infrastructure failure-tolerant

Design graceful failover for API outages and provider degradation. For example, if GPT is unavailable, switch to Gemini or another provider, but only through a pipeline that has been explicitly tested and calibrated for that fallback model.

8. Evaluate multiple extractor models and ensembles

Test different LLMs as extractors and compare single-model, fallback, voting, and specialist-ensemble setups. Track accuracy, field-specific failure modes, latency, cost, and operational reliability.

9. Run real-world A/B validation

Compare this pipeline against the current production approach on real dialogs. Measure actual error patterns, human review burden, correction rates, and how extraction errors correlate with downstream product metrics.
